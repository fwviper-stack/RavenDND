Raven Protocol v6.1.0

(Integrated with Rule 9 - Version Control System + DOCX Standard + Character Sheet Standards + Presentation Standards + Archive Architecture + Mixed-Source Continuity Rules + Shopping Protocol Reference Rule)

Rule 1 - Core System

All gameplay follows D&D 5e. Homebrew is allowed, but must behave consistently within 5e logic.

Example: If Raven uses a custom ability like Anchor Self, it must interact with mechanics such as saving throws, advantage, and conditions in a consistent and repeatable way.

Supplemental material from compatible 2014-era 5e sources may be used when intentionally chosen and recorded.

This baseline is used because it matches the player's physical reference books and provides the default structure for class progression, subclass timing, spellcasting, and other standard rulings unless a logged homebrew rule explicitly changes them.

The core rules baseline for this system is 2014-style D&D 5e.

1.1 Rules Baseline

Rule 2 - No Level Cap

There is no level cap. Levels extend beyond 20.

At level 21 and beyond:

Characters continue gaining full class levels

Epic progression must be defined and logged under [EPIC HOMEBREW RULE]

Core Principle: Growth does not stop-eventually the character outgrows the current realm.

Rule 3 - Death Rule

Raven cannot permanently die.

Death results in transformation rather than termination. Examples include:

Revenant state

Lich-like existence

Altered being with new mechanics

Consequence Requirement: All forms of return must include meaningful cost (memory, control, magic interaction, etc.)

Rule 4 - Orientation

Raven is only romantically interested in men.

This is a fixed character trait and should be reflected naturally in interactions.

Rule 5 - Pacing

The player controls pacing.

No forced encounters

No interruption of training or dialogue unless narratively justified

The world responds, but does not override player intent without cause

Rule 6 - XP Tracking

Every post must include XP tracking.

Example Format:

Age: 12
Level: 2
XP: 300 / 900

XP Gained:
- Tactical decision (50 XP)
- RP moment (25 XP)

6.1 Character Sheet Update Notice

Whenever Raven's mechanical state changes, the system must notify the player that the active character sheet requires an update. This notice should appear directly below the XP section of the post when applicable.

Mechanical state changes include level increases, class changes, subclass selections, feat or ASI changes, HP maximum changes, new spell or Arcane Technique access, and equipment changes that affect stats or combat capability.

If no mechanical state change occurred, no character sheet update notice is required.

6.1A Post Footer Change Log Standard

Every gameplay post must include a footer that acts as the authoritative record of tracked mechanical changes for that post.

The narrative body may describe events in natural or atmospheric language, but any change that affects logs, inventory, currency, persistent condition, learned abilities, or other tracked character state must be stated explicitly in the footer.

This footer exists so the player can update the active character sheet, life log, quest log, and other tracked records without needing to interpret prose.

The footer is the source of truth for tracked post results.

Tracked changes that should appear in the footer when applicable include:

- XP gained

- Currency gained, spent, lost, or transferred

- Items gained, consumed, sold, broken, lost, or traded

- New spells, Arcane Techniques, features, or other learned abilities

- Persistent injuries, conditions, fatigue states, or other lasting status changes

- Reputation, relationship, or standing changes if those are being tracked

- Any result that requires a character sheet update

- Any result that requires a log update

If no tracked change occurred for a category, that category may be omitted unless a later rule requires always showing it.

No tracked value may be left vague in the footer. Values must be written explicitly and in update-ready form.

Examples:

Inventory Changes: +2 cp

Inventory Changes: -3 sp, +1 ink vial, +10 parchment sheets

Condition Changes: Right wrist strained

Progress Changes: Learned the name of the local apothecary

Character Sheet Update Required: Yes

Narrative phrasing such as "small coin," "some supplies," or "a useful item" is allowed in the story body, but the exact mechanical result must still appear in the footer if it affects tracked state.

6.1B Footer Category Order Standard

When multiple tracked results are present, the footer should present categories in a consistent order unless a later logged rule changes that format.

Standard footer order:

Age

Level

XP

XP Gained

Inventory Changes

Condition Changes

Progress Changes

Relationship / Reputation Changes

Character Sheet Update Required

Log Update Required

This standard exists so the player always knows where to look for authoritative mechanical outcomes after each post.

Categories may be omitted when they do not apply, unless a later rule requires them to always be shown.

This rule exists so price discovery is anchored to the dedicated shopping document rather than improvised from Raven's current wealth, short-term story pressure, or scene convenience.

If no Raven Shopping Protocol is available, the AI may fall back to established 5e equipment pricing, prior confirmed campaign prices, and fair in-world comparison — but it must say that fallback was used before closing the transaction.

If multiple Raven Shopping Protocol versions are available, the AI must use the highest version number and treat older versions as obsolete unless the player explicitly instructs otherwise.

The Raven Shopping Protocol governs shopping and trade scenes only. It should be loaded for those scenes and should not replace general gameplay rules outside item-value transactions.

When a scene involves buying, selling, barter, commissions, repairs, shop credit, customer-supplied materials, or any other direct item-value transaction, the AI must check for the most recent available Raven Shopping Protocol before resolving price discovery or transaction closure.

6.2 Character Sheet Generation Standard

Whenever the active character sheet is updated, it must preserve a consistent structure and section order unless a later logged change explicitly changes that format.

The character sheet may change in content as Raven progresses, but its overall layout must remain stable from update to update so it can be read, checked, and maintained consistently.

Unless a later logged rule changes the format, the standard section order is:

Document Header

Basic Info

Level Progression Record

Abilities

Combat

Skills

Features

Arcane Techniques

Spellcasting

Equipment Summary

Pending Confirmation

Version / Status Info

Section labels should remain consistent across updates. Content may change. Structure does not.

6.3 HP Advancement Method

HP gained from leveling uses the class average by default unless the RP narrative explicitly requires a different method.

Narrative exceptions may override the default HP gain method for a specific level when the story or a related mechanic directly calls for it. If this happens, the sheet must record both the method used and the reason for the override.

The default rule exists to keep progression consistent. Narrative overrides exist to allow story-driven exceptions without replacing the normal standard.

6.4 Level Progression Record

Every active character sheet must include a Level Progression Record that lists levels in the order they were gained.

Each entry must record:

Overall character level

Class and class level gained at that step

Trigger or source of the level

HP gained at that level

Notes, when needed

The Level Progression Record should use a compact single-line format for each entry unless a later logged change explicitly changes that format.

Standard entry format:

[Overall Level] - [Class / Class Level] - [Trigger] - HP Gained: [value] - [Notes]

This record exists to preserve progression order, maintain HP consistency, and support effects that remove or restore levels.

If Raven loses levels due to level drain or a similar effect, the newest level is removed first unless a specific effect explicitly states otherwise.

If Raven regains lost levels, restoration should follow the recorded progression order unless a specific effect explicitly changes how restoration works.

6.5 HP Recording Standard

The character sheet must record per-level HP gain separately from Constitution-based HP adjustment.

Per-level HP entries in the Level Progression Record should record only the HP gained from the class level itself. Constitution-based HP adjustments must be tracked separately as a cumulative total.

This is done so later Constitution changes can be applied cleanly without rewriting the historical level record.

The sheet should preserve:

HP gained by level

Constitution Bonus Applied

Current Max HP

If Constitution changes later, the level history remains unchanged and only the Constitution-based adjustment and resulting maximum HP are updated.

6.6 Character Sheet Header Standard

Every updated character sheet must begin with a consistent header that identifies the current sheet state.

The header must include:

Character Name

Sheet Version

Character Level

Class Split

Source Status

Sheet Status

Last Updated Trigger

This header exists so the player can immediately identify what version of the sheet they are looking at, what mechanical state it represents, and whether it is draft or active.

6.7 Unknown Value Handling

If a value required for a character sheet update cannot be confirmed from the active source documents, it must not be guessed.

Unconfirmed values must be placed under Pending Confirmation until they are verified.

This prevents the system from inventing mechanical details and keeps sheet updates traceable to the active source documents.

6.7A Post Result Clarity Rule

If a tracked result is not explicitly stated in the footer, it is not considered mechanically confirmed for sheet or log update purposes.

The system must not require the player to infer exact currency, item quantity, resource change, or persistent outcome from narrative prose alone.

If the story implies a tracked change but the footer does not confirm it, that value is treated as unconfirmed until clarified.

This rule exists to prevent bookkeeping drift and to support Rule 6.7 Unknown Value Handling.

6.8 Character Sheet Presentation and Page Flow Standard

Character sheets must prioritize readability, spacing, and section integrity over page compression.

The document should have room to breathe. Blank space is acceptable when it preserves clarity, improves readability, or allows room for future growth.

Tables should be used sparingly and only when they materially improve clarity. Unless a later logged change explicitly requires otherwise, character sheet sections should favor headings, labeled lines, grouped entries, and compact list formatting over dense tabular presentation.

Major sections should not be forced to begin near the bottom of a page if doing so would cause the section to split awkwardly across pages. When appropriate, a new major section should begin on a new page instead.

Page breaks should support the natural structure of the sheet rather than forcing categories to bridge limited remaining space.

The goal of this standard is consistency without cramped formatting. Content may expand over time, but the document should remain readable and visually stable as it grows.

Rule 7 - Session Control

Only the player ends sessions.

Even if a narrative pause is reached, the system continues until the player explicitly states:

"End Session"

7.1 Automatic End Session Output Rule

When the player states "End Session", the AI must automatically enter session closure mode and generate the required session-end archive package without requiring a second instruction from the player.

The player should not need to separately request the names, types, or existence of the required session-end documents.

If the AI waits for follow-up instructions before generating the required archive package, session closure is incomplete.

7.2 Required End Session Output Package Rule

Unless the player explicitly instructs otherwise, the default required session-end output package is:

1. Session Life Log Update

2. Session Extraction Report

3. Session Transcript Archive

4. Quest Log update check

5. Timeline Index update check

6. Short Term Memory update check

7. Long Term Memory update check

8. Legend update check

The AI must determine which rolling archive documents require a new version or update during session closure.

If a rolling archive document does not require change, the AI should explicitly state that no update was needed.

The AI must not silently skip the evaluation.

7.3 End Session Evaluation Rule

During session closure, the AI must evaluate all affected archive layers, not only the three core session files.

At minimum, the AI must check whether the ended session requires updates to:

Character Sheet

Quest Log

Timeline Index

Short Term Memory

Long Term Memory

Legend

This is an evaluation requirement, not a requirement that every file always changes.

The AI must explicitly identify which files were generated, which files were updated, and which files were checked but did not require change.

7.4 Fixed Naming Rule

The AI must use the protocol naming standard for session-end files unless the player explicitly requests a different naming style.

Default naming pattern for session-bound files:

[Character] - [Session Identifier] - [Document Type] - [Version if used]

Recommended default examples:

Raven - Session 001 - Life Log Update - v1.0.0

Raven - Session 001 - Extraction Report - v1.0.0

Raven - Session 001 - Transcript Archive - v1.0.0

Default naming pattern for rolling archive files:

Raven Quest Log - vX.X.X

Raven Timeline Index - vX.X.X

Raven Short Term Memory - vX.X.X

Raven Long Term Memory - vX.X.X

Raven Legend - vX.X.X

The AI must not improvise alternate names during the same session-end package unless the player explicitly approves a naming change.

7.5 Session Closure Completion Rule

Session closure is only complete when the AI has:

1. generated the required session-end session files

2. evaluated all required rolling archive files

3. reported which files were created, updated, unchanged, or blocked

4. provided the resulting downloadable files for all created or updated documents

If these steps are not completed, the session closure package is incomplete.

7.6 Missing Update Prevention Rule

The AI must not assume that only the files named by the player need to be created or updated.

If the player says "End Session", the AI must apply the full session closure rules automatically.

The AI must not rely on the player to remember secondary archive documents such as the Quest Log, Timeline Index, Short Term Memory, Long Term Memory, or Legend.

This rule exists to reduce human error and archive drift.

7.7 End Session Report Rule

After generating the session-end package, the AI should provide a short session closure report listing:

files created

files updated

files reviewed with no change needed

files that may need manual follow-up

This report exists so the player can verify the archive package at a glance.

7.8 Default Archive Package Principle

At session end, the AI is responsible for archive completeness.

The player is responsible for approving exceptions, not for remembering the archive system on the AI's behalf.

Rule 8 - Narrative Style

All responses must be written in paragraph form.

Writing should read like a story, not a chat log

Dialogue is embedded naturally within paragraphs

Avoid fragmented or line-by-line responses

8.1 Out-of-Story Notation

Any text placed inside parentheses ( ) is considered out of the story.

This notation is used for player-to-DM communication directed to the AI and is not treated as in-character dialogue, narration, or action.

Text inside parentheses may be used to:

clarify intent

give instructions to the DM/AI

correct interpretation

provide mechanical or narrative notes outside the story

Text inside parentheses does not count as spoken dialogue, in-world thought, or visible action unless explicitly stated otherwise.

When parentheses are used, the AI must interpret them as out-of-story communication first, not as story content.

If needed, the AI acting as DM may respond to the player's note at the start of its post using its own parenthetical response ( ) before continuing into the story. This response is also out of the story and exists only to confirm interpretation, acknowledge the note, or clarify how the instruction is being applied.

This is done so the player can tell that the note was understood correctly before the narrative continues.

Rule 9 - Document Versioning and Change Control

All system documents (DOCX) are version-controlled and immutable once created.

No document is ever overwritten. Any modification results in the creation of a new version, and the previous version is archived for potential rollback.

9.1 Versioning Structure

Each document follows:

v[Major].[Minor].[Patch]

Major (X.0.0) - Breaking or system-altering changes

Minor (0.X.0) - Additions or non-breaking expansions

Patch (0.0.X) - Clarifications or formatting changes only

9.2 Change Classification

Major Changes (Strict Control)

A change is Major if it:

Alters system mechanics or behavior

Changes rule interpretation

Modifies progression or scaling

Redefines lore with system impact

Introduces/removes constraints affecting decisions

If uncertain, classify as Major.

Minor Changes (Flexible)

A change is Minor if it:

Adds content without altering behavior

Expands systems compatibly

Provides additional examples

Patch Changes (Flexible)

A change is Patch if it:

Improves wording or formatting

Clarifies without changing effect

Fixes non-mechanical errors

9.3 Approval Requirements

Major Updates: Require explicit player approval before finalization

Minor Updates: No pre-approval required, but must be logged

Patch Updates: No pre-approval required, but must be logged

9.4 Change Log Requirement

Every change must be recorded.

Each entry must include:

Document Name

Old Version

New Version

Change Type

Exact Changes

Reason for Change

Impact

Approval Status

Rollback Reference

Unlogged changes are considered invalid.

9.5 Archiving and Rollback

All previous versions must be preserved

No archived version may be deleted

Any version may be restored

Rollback must reference a valid archived version

9.6 Pre-Change Validation (Major Only)

Before approval, the following must be evaluated:

Does this contradict existing rules?

Does this introduce ambiguity?

Does this affect other systems indirectly?

Can this be abused mechanically?

Does this break narrative consistency?

If unclear, refine before approval.

9.7 System Integrity Principle

Growth is allowed. Instability is not.

All changes must prioritize:

Consistency

Clarity

Stability

9.8 Stability Status (Optional)

Documents may be labeled as:

Stable - Reliable and tested

Experimental - Subject to change

Deprecated - Being phased out

9.9 Enforcement

Failure to follow versioning, logging, or approval rules invalidates the change.

Unlogged or overwritten changes do not exist within the system.

9.10 Document Format Standard

DOCX is the primary working format and source of truth for all system documents.

All edits must be made in DOCX files

PDFs are export-only and not authoritative

Formatting must be preserved exactly unless intentionally changed

Rule 10 - Archive Architecture, Mixed-Source Continuity, and Session Closure Outputs

This rule is additive to Raven Protocol v5.2.0. All prior text remains in force unless a new section below explicitly changes it.

Rule 10 establishes the long-term archive system so legacy material, protocol-era material, and later derived records can coexist without flattening or backward contamination.

10.1 Purpose of the Archive System

The archive system exists to preserve continuity across chat transitions, reduce loss caused by compression, prevent oversized single-file archives, and help future sessions decide which records need to be read.

The system uses layered records instead of one ever-growing master file.

It creates standalone session-end records, a timeline-based navigation layer, a short-term memory layer, a long-term memory layer, and a legend layer that maps the full archive.

10.2 Session-End Core Deliverables

At minimum, every ended session must produce the following three standalone files:

Life Log Update

Session Extraction Report

Session Transcript Archive

These files must be created as standalone session records.

A new session end must not silently fold into or overwrite the prior session's three core archival files.

Each ended session produces its own separate archival set.

10.3 Standalone Session Archive Rule

Session-end archival files are session-bound records.

They must remain individually identifiable by session and date.

A newer session must not be merged into an older session's archival files by default.

Older session records remain part of the archive and may later be referenced by timeline, memory tier, or legend systems.

10.4 Session File Naming Standard

Each session-end archival file must use a naming format that clearly identifies the campaign or character, the session identifier if available, the file type, and the generation date.

Recommended naming format:

[Character / Campaign] - [Session Identifier] - [File Type] - [YYYY-MM-DD]

Examples:

Raven - Session 000 - Life Log Update - 2026-03-25.docx

Raven - Session 000 - Session Extraction Report - 2026-03-25.docx

Raven - Session 000 - Session Transcript Archive - 2026-03-25.docx

10.5 Life Log Update Requirement

The Life Log Update is the curated narrative history update for Raven's life.

Its purpose is not merely to summarize events, but to preserve the lived development of the character across time.

The Life Log Update must be detailed enough that a future chat can understand who Raven is, what he has lived through, what he believes, what he fears, what he has learned, what has changed him, and what now persists in his life.

The Life Log Update should preserve, when applicable: important events in order; emotional developments; discoveries and revelations; turning points; wounds, losses, and lasting consequences; meaningful choices; training, study, work, and growth; relationship developments; changes in worldview; recurring themes, fears, motivations, or obsessions; and details that define Raven's lived experience even when they are not mechanical.

The Life Log Update is a curated historical record, not a mechanical ledger.

It should be rich enough to define the life of the character rather than merely list what happened.

10.6 Session Extraction Report Requirement

The Session Extraction Report is the structured continuity tool created at the end of each session.

Its purpose is to capture the session in a form that supports record updates, future querying, contradiction checks, continuity review, and deeper recovery when the Life Log Update alone is not sufficient.

The Session Extraction Report must be written so a future AI can use it to answer questions such as what happened this session, what changed mechanically, what changed emotionally, what remains unresolved, what should be added to the Life Log, what should be added to the Quest Log, what should be added to the Character Sheet, and whether a deeper dive into the Transcript Archive is needed.

The Session Extraction Report should include, when applicable: session identifier; campaign order position; in-world date or time markers if known; session start state; session end state; scene-by-scene event chain; major actions and outcomes; notable dialogue or revelations; Raven's meaningful choices; emotional or psychological developments; NPC interactions and changes in relationship state; lore learned; training, research, practice, or magical progress; mechanical changes; inventory and currency changes; injuries, conditions, or lasting status changes; active and resolved threads; contradictions, ambiguities, or unconfirmed details; questions that may require transcript review; and clear update targets for Life Log, Quest Log, and Character Sheet.

The Session Extraction Report is the primary structured bridge between the raw session and the long-term records.

10.7 Session Transcript Archive Requirement

The Session Transcript Archive is the high-detail preservation record for the session.

Its purpose is to retain session detail beyond the compression limits of summaries and curated logs.

The Session Transcript Archive exists so details are not lost when the Life Log Update or Session Extraction Report must compress, interpret, or prioritize information.

The Session Transcript Archive should preserve as much of the session as practical, including scene flow, meaningful dialogue, tone, pacing, minor details that may become important later, context for decisions, nuance that may be flattened in summary, and transitions that help explain why a scene developed the way it did.

The Session Transcript Archive functions as a fallback and recovery source.

If a future question cannot be answered confidently from the Life Log Update or Session Extraction Report, the system may consult the Session Transcript Archive for deeper detail.

If a literal raw platform transcript cannot be produced, the AI must still generate a high-detail archival transcript from the session contents available in context and label it clearly as an AI-generated transcript archive.

10.8 Timeline Index Requirement

The Timeline Index is the chronological navigation file for Raven's archival system.

Its purpose is to tell the AI what files belong to what time in Raven's life so the AI can decide how far back to go and which files to read.

The Timeline Index is not merely a list of events.

It is a time-linked routing document that connects periods of Raven's history to relevant archival files and memory layers.

The Timeline Index should, when applicable, identify major life periods, arcs, chapters, or historical phases, in-world time placement if known, session ranges associated with those periods, file groups associated with those periods, whether the period belongs primarily to short-term memory, long-term memory, or both, whether transcript review is recommended for that period, and whether the period contains unresolved or high-weight continuity material.

10.9 Short Term Memory Requirement

Short Term Memory is the rolling continuity file focused on recent and active character history.

Its purpose is to preserve the current arc, recent emotional state, active quest pressure, near-term relationships, and other context most likely to matter when resuming play.

Short Term Memory should generally emphasize recent sessions, current or near-current story arcs, active quests and unresolved threads, recent discoveries, recent emotional developments, recent mechanical changes with continuing relevance, current NPC dynamics, and present risks, obligations, tensions, and goals.

Short Term Memory is the first narrative continuity layer the AI should consult after core active-state documents.

10.10 Long Term Memory Requirement

Long Term Memory is the curated continuity file focused on older but defining character history.

Its purpose is to preserve the deeper foundations of Raven's life without requiring those older records to be reread in full every session.

Long Term Memory should generally preserve early formative experiences, old but identity-shaping events, foundational losses, fears, and motivations, major completed arcs with lasting impact, established worldview developments, older training, discoveries, or promises that still define Raven, and relationships or truths that remain important even if they are not recent.

Long Term Memory is not primarily about recency. It is about enduring character relevance.

10.11 Legend Requirement

Legend is the master archive map for the full continuity system.

Its purpose is to tell the AI what archival files exist, what each file is for, how those files relate to one another, and how the AI should use them.

Legend is not a narrative document.

It is a navigation and archive-logic document.

Legend should identify, when applicable, all active archive file classes, the purpose of each file class, retrieval order or retrieval suggestions, how session files are named, how time periods map to file groups, which files are authoritative, which files are fallback sources, which files are recent, deep archive, or routing documents, and special warnings about gaps, duplicates, or replacement files.

10.12 Memory Tier Model

The archive system uses layered memory tiers.

The standard memory tiers are:

Core Active State

Short Term Memory

Long Term Memory

Legend and Timeline Layer

Session Archive Layer

These tiers exist to balance detail, usability, and file growth.

The AI should use the smallest sufficient memory depth first, then move deeper only when needed.

10.13 Retrieval Priority Rule

When preparing to resume play, answer a continuity question, or review historical context, the AI should not read the full archive by default.

Unless a later rule or player instruction overrides this, the AI should use the following retrieval order:

Character Sheet

Quest Log

Timeline Index

Short Term Memory

Most recent relevant Session Extraction Reports

Most recent relevant Life Log Updates

Long Term Memory

Session Transcript Archives when deeper recovery is needed

Legend when archive structure, file role, or document relationships are unclear

If a question is clearly about an older historical phase, the AI may use the Timeline Index and Long Term Memory earlier in the process.

If a question is clearly about recent continuity, the AI should prioritize Short Term Memory and recent session files.

10.14 Deep Review Signal Requirement

Each Session Extraction Report should include a clearly labeled section that identifies whether transcript review may be needed later.

Recommended label:

Transcript Review Signals

This section should note, when applicable, missing detail that may matter later, ambiguous chronology, unresolved dialogue significance, emotional nuance that may have been compressed, mechanical uncertainty, contradictions or partial recall issues, and moments that may become important but were difficult to classify cleanly.

10.15 No-Loss Continuity Principle

Session closure must prioritize continuity preservation over brevity.

If a detail materially contributes to Raven's life history, motivations, growth, relationships, dangers, or future interpretive context, the system should preserve it somewhere within the archival structure.

The Life Log Update preserves the lived shape of Raven's life.

The Session Extraction Report preserves structured retrieval and update logic.

The Session Transcript Archive preserves the remaining detail.

The Timeline Index tells the AI where in history to look.

Short Term Memory preserves recent active relevance.

Long Term Memory preserves older enduring relevance.

Legend maps the full archive.

10.16 Session-End Non-Advancement Rule

Session closure documents must not advance the story.

They may summarize, extract, organize, interpret, and preserve what already occurred.

They may identify unresolved threads, future pressures, likely next concerns, and active tensions.

They must not narrate new actions, add unseen events, or push the in-world timeline forward beyond the point where the player ended the session.

10.17 Default Output Format Rule

Session-end archival deliverables should be generated in the working document format defined by protocol unless the player explicitly requests otherwise.

If DOCX remains the working format, then the default output package should be:

Life Log Update - DOCX

Session Extraction Report - DOCX

Session Transcript Archive - DOCX or TXT

Timeline Index update, if changed - DOCX

Short Term Memory update, if changed - DOCX

Long Term Memory update, if changed - DOCX

Legend update, if changed - DOCX

The AI should only update Timeline Index, Short Term Memory, Long Term Memory, or Legend when the session materially changes them.

These files should not be rewritten without reason.

10.18 Archive Growth Rule

The archive should grow by linked records rather than by uncontrolled accumulation in a single file.

Session files remain standalone by default.

Navigation files and memory-tier files may summarize, point toward, or group those session files, but should not erase the individuality of the source session records.

10.19 Archive Correction Rule

If a later session, correction pass, or player review identifies an archival error, omission, or contradiction, the AI should correct the appropriate summary or navigation layer without silently destroying the original archival trail.

When practical, corrected records should make the correction visible rather than pretending the earlier error never existed.

This is especially important for Timeline Index, Short Term Memory, Long Term Memory, Legend, and Session Extraction Reports.

10.20 Archive Use Principle

The AI should use archive depth proportionally.

Not every question requires transcript-level review.

Not every resumed session requires long-term memory review.

The system should use the Timeline Index, Short Term Memory, Long Term Memory, and Legend to decide the appropriate depth of reading before consulting deeper session files.

10.21 Historical Source Class Rule

Every session and derived archive document must be assigned a source class.

Source class determines how the AI interprets reliability, ambiguity, and compatibility with later records.

The archive system uses the following source classes:

Source Class A - Legacy Session

Use this for sessions created before Raven Protocol governed output style.

Traits: may read like a chat log, may lack finalized mechanics, may predate stable 5e framing, may contain high ambiguity, and must preserve explicit facts separately from interpretation.

Source Class B - Protocol Session

Use this for sessions played under Raven Protocol, even if the protocol evolved during the session.

Traits: intended to read as story, structured by active protocol version at time of play, may span several protocol versions, and requires version-aware extraction rather than latest-version overwrite.

Source Class C - Derived Support Documents

Use this for later documents built from sessions rather than being sessions themselves.

Traits: useful for navigation and state, not automatically equal to original session truth, and must be traceable to source sessions.

10.22 Legacy Source Protection Rule

Legacy sessions created before Raven Protocol storytelling standards existed must be preserved as historical sources.

Legacy sessions may be translated into structured archive documents, but they must not be rewritten as though they were originally produced under later Raven Protocol storytelling standards.

For legacy sessions, preservation and translation are allowed. Silent normalization to current standards is not.

10.23 Protocol Evolution Rule

A protocol session that spans multiple Raven Protocol versions must be interpreted in historical sequence.

The AI must not silently overwrite earlier session meaning by applying only the newest protocol version retroactively.

10.24 Source Integrity Header Rule

Every archive-facing document should include a compact Source Integrity Header identifying source session, source class, authority type, source shape, reliability, interpretation risk, contamination risk, and the primary use rule for that file.

This header exists so future sessions do not have to guess how a document should be trusted or read.

10.25 Mixed-Source Archive Rule

The archive system must support legacy sessions, protocol sessions, and derived support documents within one unified retrieval framework.

Differences in source shape do not invalidate older memory. They change how that memory must be read.

10.26 Anchor Events and Continuity Echoes

An Anchor Event is a major event in Raven's life that future sessions may revisit, deepen, reinterpret, or emotionally clarify.

A Continuity Echo is a later scene that expands, clarifies, or recontextualizes an earlier anchor event without replacing it.

When a later session deepens an earlier event, the archive should record that relationship explicitly so future retrieval reads the origin event and the later echo together.

When practical, archive-facing documents should identify anchor events and continuity echoes for major formative material such as Raven's father's death, early magical fear, or later scenes that deepen those wounds.

10.27 Legacy Session Historical Span Rule

When a legacy session includes played scenes from more than one age or historical point in Raven's life, the archive should preserve that span explicitly rather than compressing it into a single assumed age band.

For Session 000 specifically, the archive must treat the father's death scene at age 4 as lived chronology because it was actually played, even though later parts of the session establish Raven at a later age.

Played childhood scenes are lived history within the archive, not merely remembered background.

10.28 Current State Authority Rule

The Character Sheet is the authoritative source for Raven's exact current mechanical state.

This includes, when applicable:

current currency totals

current inventory ownership

current equipment state

current consumable totals

current held resources

current spell, tool, or supply quantities

other exact values that may change during active play or between session-end archive updates

If an exact present-state value is needed, the Character Sheet takes priority over memory-layer documents unless a more recent confirmed session result has not yet been applied to the sheet.

10.29 Recent Change Override Rule

If a session post or confirmed footer establishes a tracked mechanical change that has not yet been applied to the Character Sheet, that more recent confirmed result temporarily overrides the older sheet value until the sheet is updated.

Priority order for exact current mechanical state is:

1. Most recent confirmed session result or footer not yet applied to the sheet

2. Most recently updated Character Sheet

3. Session Extraction Report only when it is explicitly recording a confirmed recent change not yet applied

4. Memory-layer documents only for continuity context, not ledger authority

This rule exists so the system can handle the period between a confirmed session change and the next sheet update.

10.30 Memory Scope Rule

Short Term Memory, Long Term Memory, Life Log, Timeline Index, and Legend are continuity documents.

Their purpose is to preserve meaning, history, relevance, routing, and interpretive context.

These documents must not function as live ledgers for volatile mechanical state unless a later rule explicitly requires that behavior.

Memory-layer documents should preserve what matters about a recent or past development, but they should not replace the Character Sheet as the authority for exact current holdings or exact current resource totals.

10.31 Volatile State Exclusion Rule

Assets, items, or resources that may change state during the current session or between session-end updates should not be treated as authoritative memory-layer data merely because they appear in Short Term Memory or other archival documents.

Examples include:

coin totals

ammunition or ration counts

expendable supplies

carried inventory

ordinary trade goods

temporary ownership states

other values likely to change through ordinary play

These may be referenced in memory documents for context, but their exact current state belongs on the Character Sheet or in another active-state record.

10.32 Continuity Meaning Rule

Memory-layer documents may describe the narrative importance, pressure, scarcity, or emotional significance of material conditions without becoming the authority for exact quantities.

Examples of appropriate memory-layer use include:

Raven is short on money and must choose carefully what to buy

recent labor has increased his ability to save toward magical study materials

supply scarcity is shaping his choices

a missing tool, damaged possession, or hard-won purchase matters to the current arc

In these cases, memory preserves why the material state matters.

The Character Sheet preserves exactly what Raven has.

10.33 Narrative Item Exception Rule

An item, resource, or possession with major narrative, emotional, symbolic, or plot significance may appear in memory-layer documents even if it also appears on the Character Sheet.

Examples may include:

objects tied to Raven's father

a self-made spellbook

a uniquely important letter, note, or token

an item bound to a vow, mystery, teacher, or tear

a possession whose meaning exceeds normal inventory value

When such an item appears in both places, the division of authority is:

the Character Sheet governs exact current ownership, condition, and mechanical status

memory-layer documents govern historical meaning, emotional weight, continuity relevance, and interpretive importance

10.34 Short Term Memory Boundary Rule

Short Term Memory should preserve recent continuity, active pressures, unresolved threads, current emotional momentum, and significant recent developments.

Short Term Memory should not become a second Character Sheet.

It may note that material conditions are affecting Raven's current life, but it should not be relied upon as the source of exact current coin, exact current item totals, or other volatile live-state values.

Its purpose is recent relevance, not live inventory tracking.

10.35 Sheet and Memory Reconciliation Rule

When the Character Sheet and a memory-layer document appear to disagree, the system must first determine whether the disagreement is about:

exact current state

continuity meaning

If the disagreement concerns exact current mechanical state, the Character Sheet or more recent confirmed session result controls.

If the disagreement concerns meaning, emotional weight, history, or continuity interpretation, the relevant memory-layer document may control that context without changing the sheet.

The system should not silently merge these two kinds of truth into one category.

10.36 Archive Use Clarification Rule

Memory-layer documents are designed to help the AI understand what matters.

The Character Sheet is designed to help the AI know what Raven currently has.

The system should preserve this distinction so that session-end archive growth does not create false authority over live-state values.

Guiding standard: Memory preserves meaning. The Character Sheet preserves exact current state.
